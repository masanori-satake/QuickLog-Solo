# QuickLog-Solo: AI Agent Guidelines (AGENTS.md)

このドキュメントは、QuickLog-Solo の設計思想、過去の重要な判断の背景（Why）、および AI エージェント（あなた）が開発・保守を行う際の具体的な行動指針をまとめたものです。

## 1. コア・フィロソフィー (Core Philosophy)

### 「1秒で記録、1秒で集計、1秒で安心」
本プロジェクトが最も大切にしているのは、ユーザーの認知負荷を最小化し、かつデータの透明性と安全性を最大限に高めることです。
また、本プロジェクトは Public なオープンソース・プロジェクトとして公開されており、不特定多数のユーザーが利用・参照することを前提としています。

### QuickLog-Solo の位置づけと提供の狙い
QuickLog-Solo は、高度な「工数管理システム」ではなく、あくまで個人のための「作業メモ（記録）ツール」です。
- **代替対象:** 手書きの開始/終了時刻メモ、スプレッドシートへの手動記録とマクロ集計といった、個人の泥臭い単純作業をデジタルで効率化することを目指しています。
- **非目標:** チーム全体の工数管理、高度な分析レポート、外部サービスとの連携などは提供の範囲外（Non-Goal）です。
- **狙い:** 「工数管理」という言葉から想起される重厚な仕組みではなく、付箋にメモを書くような気軽さで作業時間を記録し、必要な時に一瞬で集計できる体験を提供することに特化しています。

- **持続可能性 (Sustainability):** 外部サーバーや API に依存せず、オフライン（ブラウザ内）で完結すること。ブラウザやタブが閉じられても、計測状態が確実に復元・継続されること。
- **ミニマリズム (Minimalism):** 「必要十分（Less is More）」。過剰な機能追加を避け、特定のワークフロー（作業記録）の摩擦を最小化すること。
- **透明性 (Transparency):** データの保存先（IndexedDB）や通信が行われないことを明示し、ユーザー（技術者）が安心して利用できること。
- **保守性・長寿命 (Maintainability & Longevity):** 外部フレームワークやライブラリを一切使用せず、標準的な Web API (Vanilla JS) のみで構築すること。

## 2. 判断の背景 (Why)

詳細な設計思想、技術選定、および判断の背景については、製品仕様書 [docs/spec.md](docs/spec.md) に集約されています。AI エージェントは常に `spec.md` を最新の正解として参照し、重要な変更を行った際は `spec.md` を更新しなければなりません。

ここでは、`spec.md` に記載されている主要な方針を要約します。

### 技術選定とプロジェクト方針
- **技術選定: Vanilla JS & フレームワーク禁止（OSS 非依存による長寿命化）:** 依存関係の複雑化とメンテナンスコストの排除。詳細は `spec.md` 2.2 参照。
- **UI システム: Material 3 (M3) Tonal Palette:** 単一のアクセントカラーではなく、トナルパレット（Primary, Secondary, Tertiary 等）を使用することで、アクセシビリティ（コントラスト）を確保しつつ、カテゴリごとの視覚的な識別性を向上させるためです。
- **body への定義:** テーマ用の CSS 変数を `:root`ではなく `body` に定義しているのは、テーマの切り替え（`.theme-dark` 等のクラス付与）時に、カスケードによって変数が確実に上書き・伝播されるようにするためです。
- **RGB コンポーネントの提供:** 半透明の背景（オーバーレイ等）を実現するため、`shared/css/m3-theme.css` では `--md-sys-color-surface-container-rgb` のように、色の RGB 成分を個別に提供しています。
- **言語設定の永続化標準 (Language Persistence Standard):** ランディングページとサブプロジェクト（Studio, Editor, Guide）間で表示言語を同期するため、URL パラメータ `?lang=` を活用します。初期化時はブラウザのロケールよりもこのパラメータを優先し、言語変更時は `window.history.replaceState` を用いて URL 状態を維持します。
- **UI 言語の標準順序:** ランディングページや設定画面の言語セレクターにおいて、ユーザーが一貫した操作性（メンタルモデル）を維持できるようにするため、1. English, 2. 日本語 ... の順序を標準としています。
- **日本語用語の標準化:** ユーザーの混乱を防ぐため、「カテゴリ」、「インポート/エクスポート」、「終了」、「(停止)」、「時刻アジャスト」、「アラーム・通知機能」といった用語を標準化しています。
### データ管理とバックアップ
- **データ管理: 40日間 & 100件表示:** パフォーマンス維持と「直近の振り返り」への特化のため。DOM 負荷を抑え、動作を軽快に保ちます。
- **ローカルファイルバックアップ (Backup):** ブラウザのキャッシュクリア等によるデータ消失から守るため。ログとカテゴリには NDJSON、設定には JSON を採用。
- **グリフの選定ポリシー:** 「Local Only」を誤解させないため、クラウド（雲）アイコンを禁止し、物理ストレージ（hard_drive 等）を採用。
- **バックアップ/リストアのデータマッピング標準:** 外部保存用データと内部状態の互換性維持のため、設定の `animation` → `defaultAnimation` 変換や、カテゴリの `order` 保存などを定義。
- **終了ロジック: `SYSTEM_CATEGORY_IDLE` & `isManualStop`:** 日報作成時の視認性向上と、手動/自動終了の判別のため。
- **重複マーカー防止:** 停止ボタン連打等による重複記録を防止。
- **日付変更時の自動記録：専用設定からアラーム機能への移行:** 柔軟性向上とコード簡素化のため。
- **履歴 CSV インポートの整合性バリデーション:** 未来時刻や重複データの混入を防ぐ厳格なバリデーション。
- **厳格なスキーマ検証の強制:** すべてのインポート/リストアにおいて、`schema.js` による厳格な検証を行い、不明なデータは拒否します。
- **文字列の永続性とローカライズ・ポリシー:** 内部 ID（`__IDLE__` 等）は不変とし、UI 表示時のみ翻訳します。

### UI インタラクションとデザイン
- **ページネーション: 8x2 グリッド & ホイール操作:** サイドパネルの限られたスペースで、マッスルメモリーによる打刻を支援。
- **カテゴリ名の重複・複製ポリシー:** 「Task (1)」のような連番を付与し、名前の衝突を回避。
- **UI レイアウトの堅牢性: 可変長テキストへの対応:** 多言語環境でのレイアウト崩れを防ぐため、`ellipsis` 等を徹底。
- **UI インタラクション: フォールスルー・パターン:** 前面テキストの操作と背景アニメーションへの干渉を両立。
- **設定画面のアイコン化とツールチップ (v0.31.0):** 水平スペース確保とレイアウト崩れ防止のため。
- **コピー操作の標準化と視認性の確保:** ユーザーの期待に沿った一貫したフィードバックを提供。
- **日報出力における経過時間フォーマット:** 視覚的なリズムによる可読性向上。
- **タイポグラフィ: 言語別フォント最適化と Web フォントの徹底:** 全言語で一貫したモダンな外観を維持。

### 背景アニメーション (Visual Healing)
- **2分周期のサイクル:** 視覚的な変化を緩やかにし、集中を妨げないため。
- **LCD ドットマトリクス・スタイル:** レトロな温かみと「1秒の重み」を表現。
- **アニメーション初期化の堅牢性:** レースコンディションや未定義アクセスを防ぐ設計。
- **モジュール化の理由:** コアロジックの肥大化を防ぎ、プラグイン形式での拡張を容易にするため。
- **安全性の確保 (Worker 化と CSP):** Web Worker 分離と CSP によるユーザーデータの保護。
- **負荷監視と初期化ラグ対策:** 低スペック環境での安定性とウォームアップ期間の確保。
- **透明なレイヤー構造による視認性確保:** 前面要素がアニメーションを隠さないよう、`!important` 指定で背景を透過。
- **視認性向上と制作ガイド:** 前面 UI と背景の関係性を直感的に理解できるよう `exclusionStrategy` を統合。
- **パフォーマンス最適化標準:** `TypedArray` と行優先インデックスによる計算効率向上。

### サブプロジェクト (Category Editor & Studio)
- **業務カテゴリ・エディタとクリップボード連携:** 俯瞰編集の容易化と、通信を行わない安全な連携。
- **インポート時の視覚的フィードバック (500ms 待機):** ローディング表示を確実に視認させ、信頼感を醸成。
- **エディタのレイアウト安定性:** 説明領域の固定高によるガタつき防止。
- **エディタのレイアウト境界:** T 字ジャンクションによる視覚的な区切り。
- **タグボックス (Tag Box):** ユニークなタグの再利用とドラッグ＆ドロップによる効率化。
- **タグ置換 (Tag Replace):** 大規模なタグ変更の一括処理。
- **業務カテゴリ・エディタのナビゲーション最適化 (v0.32.0):** Vercel 移行に伴う迅速な改善提供と、一貫した UX。
- **エディタのテスト容易性標準 (window.state):** Playwright 等からの直接的な状態操作を可能に。
- **QL-Animation Studio (β版):** アニメーション提供の促進とサンドボックス環境。
- **Studio UI の視認性と開発体験の最適化:** 単一の開発パス（サンプルベース）への集中。
- **コードパーサーの最適化:** エクスポート時の重複定義防止。

### インフラ・運用・品質
- **ナビゲーションとポータビリティ: パス・ポータビリティ標準:** ローカル、Vercel、拡張機能の全環境での動作保証。
- **サイドパネル UI 起動の同期性:** 通知クリックなどのイベントから `chrome.sidePanel.open` を呼び出す際は、ブラウザの「ユーザー・ジェスチャー」制限を回避するため、非同期処理（`await` 等）の前に、イベントリスナー内で直ちに同期的に実行しなければなりません。イベントハンドラ内で非同期処理を挟むと、ブラウザが「ユーザーの直接操作（ジェスチャー）」との関連を喪失したと判断し、セキュリティ上の制限からサイドパネルが開かなくなります。
- **Vercel デプロイ制限への配慮:** `paths` フィルタによる不要なデプロイのスキップ。
- **二系統ビルド戦略 (Dual-Build Strategy):** Release版（青）と Dev版（橙）の明確な区別。
- **Web デプロイ・ワークフローのトリガー:** 依存ファイルの変更を確実にデプロイに反映。
- **診断用統計情報の提供 (Diagnostic Statistics UI):** 不具合報告時の正確な情報収集。
- **GitHub Issue テンプレートの整備:** 報告の質の向上と迅速な特定。
- **Mermaid 図のシンタックス: ユニークな参加者 ID の確保:** GitHub でのレンダリング不具合防止。
- **CSS 詳細度標準 (Stylelint):** `no-descending-specificity` 遵守による意図しないスタイルの上書き防止。
- **E2E テストの構造化:** 機能別分割と固定時刻の使用による安定性確保。
- **CSS ドキュメンテーション・ポリシー:** `!important` の使用理由の明記。
- **Release版:** 青色アイコン。`devOnly: true` が設定されたアニメーションモジュールは物理的に除外され、パッケージサイズを最小化します。Chrome / Edge 用のみを生成します。
- **Dev版:** オレンジ色アイコン。すべての開発用アニメーションを含み、マニフェスト名に `(Dev)` サフィックスが付与されます。Chrome / Edge 用のみを生成します。

### 文字列の永続性とローカライズ・ポリシー
- **理由:** IndexedDB に保存されるシステム予約文字列（`SYSTEM_CATEGORY_IDLE` 等）は、将来的なデータの互換性を維持するために変更してはなりません。また、多言語対応を円滑にするため、特定の言語に依存しない値（例: `__IDLE__`）を内部的な ID として採用し、UI 表示の際のみ翻訳関数（`t()`）を通してローカライズします。

### UI レイアウトの堅牢性: 可変長テキストへの対応
- **理由:** 多言語対応に伴い、ボタンやラベルのテキスト長が大きく変動します。レイアウト崩れを防ぐため、重要な UI 要素には `min-width: 0`, `overflow: hidden`, `text-overflow: ellipsis` を適用し、どのような言語でも「壊れない」UI を維持します。

### UI インタラクション: フォールスルー・パターン
- **理由:** 背景アニメーションへの干渉（Cats への給餌クリック等）と、前面テキストの選択・操作を両立させるためです。タイマー表示コンテナ（`#current-task-display-base`）には `pointer-events: none` を設定してクリックを背後のキャンバスに透過させ、操作が必要なテキストやラベルのみ `pointer-events: auto` で個別に有効化しています。

### テスト容易性: E2E セレクターの具体性
- **理由:** モダルの閉じるボタン等が複数の機能で共通のクラス（`.close-btn`）を持つと、E2E テスト（Playwright）において誤った要素を操作するリスク（Flaky Test）が生じます。そのため、機能ごとに固有のクラス（`.report-close-btn`, `.tag-aggregation-close-btn` 等）を付与し、テストの安定性を高めています。

### セキュリティ: 言語パラメータのサニタイズ (`getSafeLang`)
- **理由:** URL パラメータや設定から読み込まれる言語コードをそのまま使用すると、不正な文字列の注入や予期せぬ動作を招く恐れがあります。ホワイトリスト形式の `getSafeLang` を通すことで、許容された言語コードのみが CSS フォントスタックやドキュメントタイトルに適用されるようにしています。

### 日報出力における経過時間フォーマット
- **理由:** スペースの有無により視覚的なリズムを生み出し、10分未満の短いタスク（`1h 5m`）を際立たせつつ、長時間のタスク（`1h15m`）をコンパクトに表示するためです。

### コピー操作の標準化と視認性の確保
- 理由: ユーザーの期待に沿った一貫したフィードバックを提供するため、日報およびタグ集計のコピー操作は「トースト通知のみ（ボタンのテキストは変えない）」に統一しています。また、ホバー時に M3 のトナル・インバージョン（Primary Container 等）を適用することで、ダークテーマを含めた全ての環境で確実な視認性と「押せる感」を両立させています。

### 設定画面のアイコン化とツールチップ (v0.31.0)
- **理由:** 設定モーダル内のタブ（一般、カテゴリ、メンテナンス等）を Material Symbols のアイコンに刷新したのは、将来的な設定項目の増加に備えて水平スペースを確保し、かつ多言語環境でのラベル長によるレイアウト崩れを防ぐためです。
- **アクセシビリティ:** アイコンのみでは機能が伝わりにくい可能性があるため、`data-i18n-title` を用いた多言語対応のツールチップを実装し、マウスオーバー時に元のラベル（General, Categories 等）を表示することで視認性と操作性を両立させています。

### 業務カテゴリ・エディタのナビゲーション最適化 (v0.32.0)
- **理由:** 業務カテゴリ・エディタをウェブベース（Vercelホスト）に移行しました。これは、拡張機能本体の更新（ストア審査）を待たずにエディタの改善を迅速に届けるためです。
- **実装:** 本体アプリの設定画面から Vercel 上の URL (`https://quick-log-solo.vercel.app/category-editor`) へ遷移させます。この際、URL パラメータ `?from=app` を付与し、エディタ側の「戻る」リンクを非表示にする制御を維持しています。これにより、拡張機能という限られたコンテキストから遷移したユーザーに対し、一貫したユーザー体験を提供します。

### Vercel デプロイ制限への配慮
- **理由:** Vercel Free プランには「24時間で100デプロイ」という制限があるためです。
- **対策:** GitHub Actions において `paths` フィルタを活用し、ドキュメントのみの更新など製品動作に影響しない変更では Vercel デプロイをスキップするように最適化しています。

### 診断用統計情報の提供 (Diagnostic Statistics UI)
- **理由:** 不具合報告時にユーザーが内部状態（ログ件数やカテゴリ数）を正確に伝えられるようにするためです。
- **実装:** `projects/app/app.html` の About タブに統計情報表示用のプレースホルダ (`#stats-log-count`, `#stats-category-count`) を追加しました。これらは `projects/app/js/app.js` の `updateAboutStats` 関数によって IndexedDB から集計・表示され、全サポート言語の `messages.js` キー (`about-stats-logs`, `about-stats-categories`) によってローカライズされます。

### GitHub Issue テンプレートの整備
- **理由:** プロジェクトの公開に伴い、報告の質を向上させ、原因特定と優先順位付けを迅速に行うためです。
- **実装:** 日英併記の GitHub Issue Forms (`.yml`) を `.github/ISSUE_TEMPLATE/` に導入しました。
  - **Bug Report:** ブラウザ/OS情報のほか、アプリ統計、バックアップ状態、Animation Studio 固有の設定値（描画モード、FG回避戦略、Workerログ）を収集する構造にしています。
  - **Feature Request:** 対象アプリ（本体/Studio）と機能カテゴリ（ロジック/UI/アニメーション/バックアップ等）を明確にするドロップダウンを設け、要求の意図を具体化させています。

### 日本語用語の標準化
- **理由:** ユーザーの混乱を防ぎ、UI とドキュメント間での一貫性を保つためです。
  - **カテゴリ:** 長音を排した表記を採用。
  - **インポート/エクスポート:** 読み込み/書き出しよりも一般的な技術用語を採用。
  - **終了:** 停止（Stop）ではなく、作業の完了を意味する「終了」を UI の中心に据えています。
  - **(停止):** 計測が停止している状態を示す標準的なラベル。
  - **時刻アジャスト:** 丸め処理を「アジャスト」と呼称し、意図的な調整であることを明示します。
  - **アラーム・通知機能:** 「終了忘れ防止」などの曖昧な表現を避け、機能の目的（指定時刻での自動操作や一日の区切りを明確にすること）を端的に表すこの用語を標準とします。

### 日付変更時の自動記録：専用設定からアラーム機能への移行
- **理由:** 以前は「日付変更時の自動停止（23:59:59）」という専用のトグル設定が存在していましたが、システムの柔軟性を高め、コードを簡素化するために、アラーム機能の一部として統合されました。
- **実装:** 新規インストール時、アラーム設定の 5 番目のスロットに「23:59」に「作業終了」を行う設定がデフォルトで有効化されます。これにより、ユーザーは必要に応じて時刻を調整したり、無効化したりすることが標準的な UI を通じて可能になりました。

### ドキュメントの役割分担
- **理由:** 対象読者（一般ユーザー、導入担当、開発者、AI）ごとに最適な情報を提供し、情報の重複と矛盾を防ぐためです。
  - **README.md:** 製品概要とインストール方法（一般ユーザー向け）。
  - **docs/spec.md:** 仕様と設計の「Why」（意思決定者・AI 向け）。
  - **docs/README_DEV.md / README_TEST.md:** 開発・テスト手順（開発者向け）。
  - **AGENTS.md:** 設計思想と行動指針（AI エージェント専用）。

### Mermaid 図のシンタックス: ユニークな参加者 ID の確保
- **理由:** GitHub の Markdown 表示において、同一ファイル内に複数の Mermaid シーケンス図が存在する場合、参加者 ID（Participant ID）が重複すると正しくレンダリングされない可能性があるためです。各図で `C_Disc`, `C_Inst`, `C_Loop` のように、用途に応じたプレフィックスやエイリアス（`AS`）を使い、グローバルにユニークな ID を使用することを徹底します。

- **CSS 詳細度標準 (Stylelint):** `projects/app/css/style.css` において `no-descending-specificity` ルールを遵守します。共通クラス（`.material-symbols-outlined` 等）をターゲットにするセレクタは、詳細度が高くなる順に並べる必要があります（例: `.alarm-confirm-label ...` の後に `#report-date-display ...` を配置）。

- **E2E テストの構造化:** Playwright テストは機能ごとに分割し、保守性を高めています（`ui_settings.spec.js`, `analysis.spec.js`, `data_io.spec.js`, `category_editor_features.spec.js` 等）。特に日付変更線やタイムゾーンの影響を避けるため、集計テストでは `12:00:00` などの固定時刻を使用します。

- **履歴 CSV インポートの整合性バリデーション:** インポート時は (1) 未来時刻の拒否 (2) 開始時刻 > 終了時刻の拒否 (3) 完全重複行のスキップ を厳格に行い、不整合なデータが DB に混入するのを防ぎます。

- **Web デプロイ・ワークフローのトリガー:** Vercel への自動デプロイを確実に実行するため、`release_web_deploy.yml` の `paths` フィルタには `releases/**`, `package-lock.json`, `tests/**`, およびワークフローファイル自体を含めています。

### タイポグラフィ: 言語別フォント最適化と Web フォントの徹底
- **理由:** OS 標準のフォントへのフォールバック（特に日本語環境での明朝体へのフォールバック等）を防ぎ、全てのサポート言語（日・英・独・西・仏・葡・韓・中）で一貫したモダンな外観を維持するためです。
- **グリフと絵文字:** `Noto Sans Symbols` と `Noto Color Emoji` を明示的に導入することで、特殊記号や絵文字の表示をプラットフォーム間で統一し、デザインの意図を正確に伝えます。
- **動的なフィルタリング:** 設定画面のフォント選択肢を選択中の言語に最適化されたものに限定することで、ユーザーが迷わず適切なフォントを選べるようにしています。

### パフォーマンス最適化標準
- **理由:** 計算負荷の高いシミュレーションにおいて、多次元配列を避け、1次元の `TypedArray` と行優先インデックス（`y * width + x`）を使用することで、キャッシュ効率の向上と GC 負荷の低減を図っています。

### CSS ドキュメンテーション・ポリシー
- **理由:** `!important` を使用する際は、必ずその理由（例: アニメーション透過性の維持）をコメントで併記することを義務付けています。これにより、不用意な削除によるデグレードを防止します。

## 3. 開発ルール (Development Rules)

AI エージェントは、以下のルールを厳格に遵守せよ。

### 言語統制 (Language Control) - 最優先事項
AI エージェントは、あらゆる外部コミュニケーションにおいて**日本語を最優先で使用し、英語の使用を最小限に留めなければならない**。これは、本プロジェクトの透明性とユーザーとの円滑な対話を確保するための最重要ルールである。

以下のすべての場面において、日本語を使用せよ：
- **ユーザーとの対話:** `request_user_input` や `message_user` を用いたチャットや返信。
- **プルリクエストの作成:** `submit` ツールに渡す `title`（タイトル）および `description`（説明文）。
- **コミットメッセージ:** Git のコミットメッセージ（`feat: ` などのキーワード以降の説明文）。
- **PR へのコメント返信:** レビューに対する回答や補足説明。

**強制チェックの実行:**
`submit` ツールを呼び出す前に、自律的に `scripts/language_check.py` を実行し、送信内容（タイトル、本文、コミットメッセージ）に日本語が含まれていることを確認せよ。日本語（ひらがな、カタカナ、漢字）が含まれていない場合、提出は拒否される。

**技術用語の扱い:**
`Vite`, `IndexedDB`, `Playwright` などの固有名詞や技術用語を英語のまま含めることは許容されるが、文脈は必ず日本語で構成すること。

### 実装の原則
- **ディレクトリ構造を維持せよ:** ソースコードは `projects/app/` 、ユーティリティは `scripts/` に配置する。
- **ルートディレクトリのクリーンネス・ポリシー:** 検証目的で作成した一時的なスクリプト（`reproduce_*.py` や `verify_*.py` 等）をルートディレクトリにコミットしてはならない。作業完了後、コミット前に必ず削除せよ。これは CI 上の `scripts/check_root_files.py` によって厳格に検査されます。
- **Vanilla JS を維持せよ:** 実行時に必要な npm パッケージの追加は一切禁止する。
- **定数化を徹底せよ:** `projects/app/js/db.js` や `projects/app/js/app.js` にあるように、マジックナンバー、DOM ID、システム文字列は必ず定数として一元管理せよ。機能に密結合した定数はそのモジュール内で、共有されるものは `utils.js` 等で定義せよ。
- **innerHTML の使用禁止:** ユーザー入力（カテゴリ名、タグ、メモ等）を含む動的なコンテンツを表示する際は、`innerHTML` を使用してはならない。代わりに `textContent` や `innerText` を使用するか、`createElement` で要素を生成し DOM API を通じて構築せよ。
- **HTML エスケープの徹底:** `utils.js` の `escapeHtml` を使用し、どうしても HTML 文字列として扱う必要がある場合は XSS を確実に防止せよ。
- **データバリデーションの強化:** 外部からのデータインポート（NDJSON, CSV）やユーザー入力に対しては、文字列長、データ形式、件数の制限を設け、リソース枯渇や予期せぬクラッシュを防止せよ。
- **データ不整合を防げ:** `initDB` 内の自動修復ロジックのように、予期せぬクラッシュや中断が発生しても、データの整合性が保たれる（二重開始を防ぐ等）ように配慮せよ。

### ドキュメントとプロセス
- **ユーザーとのコミュニケーションおよび PR 連絡は日本語で行え:** (「言語統制」セクションを参照。ユーザーとの対話、PR、コミットメッセージ等はすべて日本語で行うこと。)
- **コミットメッセージ規約:** [Conventional Commits](https://www.conventionalcommits.org/) に準拠せよ（例: `feat: 日本語メッセージ`, `fix: 日本語メッセージ`）。コロン（:）の直後に半角スペースを入れ、その後に日本語を続ける形式を標準とする。
- **Why を記録せよ:** 重要な変更を行う際は、単に「何を変えたか」だけでなく、「なぜその方法を選んだか」を AGENTS.md の「判断の背景」に追記せよ。
- **バージョン管理とリリース:**
    - `projects/app/version.json`, `package.json`, `projects/app/manifest.chrome.json` のバージョン番号は常に一致させよ。
    - バージョン更新は `npm run version:bump` (`scripts/bump_version.py`) を使用して自動で行え。
    - CI 上の `scripts/verify_version_impact.py` により、変更内容（コミットタイプ）に応じた適切なバージョンアップが行われているか検証される。
    - **自動リリース:** `v*.*.*` タグがプッシュされると、GitHub Actions (`release_extension_packages.yml`) が起動し、自動的に GitHub Releases が作成され、Chrome 用の ZIP パッケージがアタッチされる。
    - **Jules による自律リリース:** ヒューマンエラー防止のため、Jules は機能追加や修正が完了した際、自律的に `npm run version:bump` を実行し、適切なタグを作成・プッシュしてリリースをトリガーすることが推奨される。

### 深い計画プロトコル (Deep Planning Protocol)
- **原則:** 実行前に「絶対的な確信」を得ること。
- **手順:**
  1. ゴールを深く理解するための対話を行う。
  2. 疑問点や前提条件を質問で確認する。
  3. 承認は質問への回答の後に行われる。
  4. `request_user_input` を活用し、計画承認後にのみ自律的な実行へ移る。

## 4. 設計原則の適用 (Design Principles)

本プロジェクトにおける各原則の体現方法は以下の通りです。

- **KISS (Keep It Simple, Stupid):** `logic.js` は DOM に依存せず、純粋なデータ処理に特化させている。
- **DRY (Don't Repeat Yourself):** UI の更新（`updateUI`）を 1 箇所に集約し、BroadcastChannel を通じた同期も最小限のロジックで行っている。
- **SLAP (Single Level of Abstraction Principle):** `app.js` のイベントハンドラ内では高度なビジネスロジックを書かず、`logic.js` や `db.js` の関数を呼び出す形式に統一している。
- **OCP (Open-Closed Principle):** カテゴリやフォント設定をデータ（配列・ストア）駆動にすることで、コードを書き換えずに設定を追加できる構造にしている。
- **YAGNI (You Ain't Gonna Need It):** 現時点で明確な要求がない機能（例: 高度なレポート機能、他サービス連携）は実装せず、コア機能のシンプルさを維持している。
