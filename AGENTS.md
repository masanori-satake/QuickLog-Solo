# QuickLog-Solo: AIエージェントおよび開発者ガイド (AGENTS.md)

このドキュメントは、QuickLog-Solo の設計思想、過去の重要な判断の背景（Why）、および AI エージェント（あなた）や開発者が開発・保守を行う際の具体的な行動指針をまとめたものです。

## 1. はじめに
本プロジェクトは Public なオープンソース・プロジェクトとして公開されており、不特定多数のユーザーが参照することを前提としています。
ユーザーの認知負荷を最小化し、データの透明性と安全性を最大限に高めることを最優先としています。

### QuickLog-Solo の位置づけ
QuickLog-Solo は高度な「工数管理システム」ではなく、あくまで個人のための「作業メモ（記録）ツール」です。
- **狙い:** 付箋にメモを書くような気軽さで作業時間を記録し、必要な時に一瞬で集計できる体験を提供すること。
- **非目標:** チーム全体の管理、高度な分析レポート、外部サービスとの連携。

## 2. 設計思想の要約
詳細な設計思想、技術選定、および判断の背景については、製品仕様書 [docs/spec.md](docs/spec.md) に集約されています。開発時は常に `spec.md` を最新の正解として参照してください。

### コア・フィロソフィーの要点
- **持続可能性 (Sustainability):** 外部サーバーに依存せず、ブラウザ内（オフライン）で完結すること。
- **ミニマリズム (Minimalism):** 必要十分な機能に絞り、記録の摩擦を最小化すること。
- **透明性 (Transparency):** データの保存先や通信の遮断を明示し、ユーザーが安心して利用できること。
- **保守性・長寿命 (Maintainability & Longevity):** 外部フレームワークを一切使用せず、標準的な Web API (Vanilla JS) のみで構築すること。

## 3. 開発・行動ルール

AI エージェントおよび開発者は、以下のルールを厳格に遵守しなければなりません。

### 3.1. 言語統制 (Language Control) - 最優先事項
あらゆる外部コミュニケーションにおいて**日本語を最優先で使用し、英語の使用を最小限に留めなければならない**。

以下のすべての場面において、日本語を使用せよ：
- **対話:** `request_user_input` や `message_user` によるチャット。
- **PR作成:** タイトル（`title`）および説明文（`description`）。
- **コミットメッセージ:** Git のコミットメッセージ（`feat: ` などのキーワード以降）。
- **コメント返信:** レビューに対する回答。

**強制チェック:** `submit` 前に `scripts/language_check.py` を実行し、日本語が含まれていることを確認せよ。

### 3.2. 公開プロジェクトとしての指針
- **無保証 (No Warranty):** 本ソフトウェアは個人による提供であり、無保証であることを README.md や About 画面で一貫して明示し続けなければならない。
- **所有権確認:** `google-site-verification` メタタグや認証ファイルを勝手に削除・変更・移動してはならない。
- **自己責任の原則:** 本ツールやドキュメントを利用したことによる損害について、開発者は責任を負わない。

### 3.3. 実装の原則
- **ディレクトリ構造:** ソースコードは `projects/app/`、ユーティリティは `scripts/` に配置。
- **ルートディレクトリのクリーンネス:** 一時的なスクリプト（`reproduce_*.py`, `verify_*.py` 等）をコミットしてはならない。CI で厳格に検査される。
- **Vanilla JS の維持:** 実行時に必要な npm パッケージの追加は一切禁止。
- **innerHTML の使用禁止:** ユーザー入力を含む動的なコンテンツには `textContent` や DOM API を使用し、XSS を確実に防止せよ。
- **定数化:** マジックナンバーや DOM ID は必ず定数（`constants.js` 等）として一元管理せよ。
- **データバリデーション:** インポートデータやユーザー入力に対し、文字列長や形式の制限を設け、クラッシュを防止せよ。

### 3.4. ドキュメントとプロセス
- **Why を記録せよ:** 重要な変更を行う際は、単に「何を変えたか」だけでなく、「なぜその方法を選んだか」を `spec.md` または `AGENTS.md` に追記せよ。
- **バージョン管理:** バージョン更新は `npm run version:bump` を使用して自動で行え。CI 上でコミットタイプに応じた適切な更新が検証される。
- **リリース:** `v*.*.*` タグのプッシュにより GitHub Actions が ZIP パッケージを自動作成する。

### 3.5. 深い計画プロトコル (Deep Planning Protocol)
- 実行前に「絶対的な確信」を得るための対話を行う。
- `request_user_input` を活用し、計画承認後にのみ自律的な実行へ移る。

## 4. 設計原則の適用 (Design Principles)
- **KISS (Keep It Simple, Stupid):** ロジックは DOM に依存させず、シンプルに保つ。
- **DRY (Don't Repeat Yourself):** UI 更新や状態同期のロジックを 1 箇所に集約する。
- **SLAP (Single Level of Abstraction Principle):** イベントハンドラ内では抽象度の高い関数呼び出しに留める。
- **OCP (Open-Closed Principle):** カテゴリや設定をデータ駆動にすることで、コードを書き換えずに拡張可能にする。
- **YAGNI (You Ain't Gonna Need It):** 現時点で明確な要求がない過剰な機能は実装しない。
