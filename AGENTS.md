# QuickLog-Solo: AI Agent Guidelines (AGENTS.md)

このドキュメントは、QuickLog-Solo の設計思想、過去の重要な判断の背景（Why）、および AI エージェント（あなた）が開発・保守を行う際の具体的な行動指針をまとめたものです。

## 1. コア・フィロソフィー (Core Philosophy)

### 「1秒で記録、1秒で集計、1秒で安心」
本プロジェクトが最も大切にしているのは、ユーザーの認知負荷を最小化し、かつデータの透明性と安全性を最大限に高めることです。
また、本プロジェクトは Public なオープンソース・プロジェクトとして公開されており、不特定多数のユーザーが利用・参照することを前提としています。

### QuickLog-Solo の位置づけと提供の狙い
QuickLog-Solo は、高度な「工数管理システム」ではなく、あくまで個人のための「作業メモ（記録）ツール」です。
- **代替対象:** 手書きの開始/終了時刻メモ、スプレッドシートへの手動記録とマクロ集計といった、個人の泥臭い単純作業をデジタルで効率化することを目指しています。
- **非目標:** チーム全体の工数管理、高度な分析レポート、外部サービスとの連携などは提供の範囲外（Non-Goal）です。
- **狙い:** 「工数管理」という言葉から想起される重厚な仕組みではなく、付箋にメモを書くような気軽さで作業時間を記録し、必要な時に一瞬で集計できる体験を提供することに特化しています。

- **持続可能性 (Sustainability):** 外部サーバーや API に依存せず、オフライン（ブラウザ内）で完結すること。ブラウザやタブが閉じられても、計測状態が確実に復元・継続されること。
- **ミニマリズム (Minimalism):** 「必要十分（Less is More）」。過剰な機能追加を避け、特定のワークフロー（作業記録）の摩擦を最小化すること。
- **透明性 (Transparency):** データの保存先（IndexedDB）や通信が行われないことを明示し、ユーザー（技術者）が安心して利用できること。
- **保守性・長寿命 (Maintainability & Longevity):** 外部フレームワークやライブラリを一切使用せず、標準的な Web API (Vanilla JS) のみで構築すること。

## 2. 判断の背景 (Why)

詳細な設計思想、技術選定、および判断の背景については、製品仕様書 [docs/spec.md](docs/spec.md) に集約されています。AI エージェントは常に `spec.md` を最新の正解として参照し、重要な変更を行った際は `spec.md` を更新しなければなりません。

ここでは、`spec.md` に記載されている主要な方針を要約します。

### 技術選定: Vanilla JS & フレームワーク禁止（OSS 非依存による長寿命化）
- **理由:** 依存関係の複雑化とメンテナンスコストの排除。詳細は `spec.md` 2.2 参照。

### UI システム: Material 3 (M3) Tonal Palette
- **理由:** 単一のアクセントカラーではなく、トナルパレット（Primary, Secondary, Tertiary 等）を使用することで、アクセシビリティ（コントラスト）を確保しつつ、カテゴリごとの視覚的な識別性を向上させるためです。
- **body への定義:** テーマ用の CSS 変数を `:root` ではなく `body` に定義しているのは、テーマの切り替え（`.theme-dark` 等のクラス付与）時に、カスケードによって変数が確実に上書き・伝播されるようにするためです。
- **RGB コンポーネントの提供:** 半透明の背景（オーバーレイ等）を実現するため、`shared/css/m3-theme.css` では `--md-sys-color-surface-container-rgb` のように、色の RGB 成分を個別に提供しています。

### データ管理: 40日間 & 100件表示
- **理由:** 「直近の振り返り」に特化し、古いデータの蓄積によるパフォーマンス低下やバックアップの肥大化を防ぐためです。40日間あれば月次の精算にも対応可能です。履歴表示を100件に絞っているのは、DOM 要素の増大による UI のもたつきを防ぎ、常にサクサクとした動作を維持するためです。

### ローカルファイルバックアップ (Backup)
- **理由:** ブラウザのキャッシュクリア等による予期せぬデータ消失から記録を守るためです。ログとカテゴリには NDJSON 形式を採用し、行ベースの追加や手動での並べ替えを容易にしています。一方、設定項目については可読性を重視し、インデントされた `settings.json` (JSON形式) を採用しています。
- **グリフの選定ポリシー:** バックアップ機能の UI において、「雲（Cloud）」を連想させるグリフの使用は禁止します。これは、QuickLog-Solo の「外部通信を一切行わない（Local Only）」という重要ポリシーに対し、ユーザーがクラウドへのデータ送信を想起し、プライバシー上の誤解を招くリスクを避けるためです。代わりに、物理的なローカルストレージを明示的にイメージさせる `hard_drive` グリフ等を使用します。

### 終了ロジック: `SYSTEM_CATEGORY_IDLE` & `isManualStop`
- **理由:** 単純な「終了時刻の記録」ではなく、あえて「終了マーカー（待機ログ）」を記録するのは、日報作成時に「いつ作業が終わったか」を視覚的に明確にするためです。また、手動終了と自動終了（次のタスク開始による終了）を区別することで、計測の連続性を保ちつつ、ユーザーの明示的な意思表示（作業完了）を履歴に残せるようにしています。
- **重複マーカー防止:** 停止ボタンの連打等により同じ時刻に複数の停止マーカーが記録されるのを防ぐため、記録前に既存ログとの重複チェックを行っています。

### ページネーション: 8x2 グリッド & ホイール操作
- **理由:** サイドパネルという限られた垂直スペースで、ボタンの大きさを維持しつつ、多数のカテゴリを管理するためです。無限スクロールではなくページ切り替えを採用したのは、ボタンの配置（座標）を固定し、筋肉の記憶（マッスルメモリー）による素早い打刻を支援するためです。

### 業務カテゴリ・エディタとクリップボード連携
- **理由:** サイドバーの狭い空間では困難だった、カテゴリの全体俯瞰や詳細な編集（タグ、色、個別アニメーション）を容易にするため、独立したフル画面エディタを導入しました。
- **データ連携の方針:** ファイル出力の手間を省きつつ、IndexedDB ハンドルを外部に露出させない安全な連携手段として、「カットバッファ（クリップボード）」経由のインポート/エクスポートを標準としました。これにより、本体アプリとエディタ間で瞬時にデータをやり取りでき、かつ「外部通信を一切行わない」ポリシーを堅持しています。

### 背景アニメーション (Visual Healing)
- **2分周期のサイクル:** 1分（60秒）ではなく2分周期にしているのは、視覚的な変化を緩やかにし、集中を妨げないためです。
- **LCD ドットマトリクス・スタイル:** 全てのキャンバスアニメーションに「LCD ドットマトリクス・スタイル（4段階のドットサイズによる低解像度表現）」を採用しているのは、最新の Material 3 デザインの中にレトロな計測機器のような温かみと、「1秒の重み」を視覚的に表現するためです。
- **モジュール化の理由:** アニメーションロジックを独立したモジュール（`shared/js/animation/*.js`）に分離し、ビルド時に自動登録する仕組みを導入したのは、コアロジックの肥大化を防ぎ、将来的なアニメーションの追加・削除を容易にするためです。これにより、コアモジュール（`app.js`）を一切修正することなく、多様な視覚表現をプラグイン形式で拡張可能になりました。
- **安全性の確保 (Worker 化と CSP):** 外部開発者が提供するモジュールの悪意の有無に関わらず、利用者のデータを守るため、アニメーションロジックを Web Worker で分離し、CSP によって外部通信を遮断しています。また、実行時間の監視を行い、負荷の高いモジュールを自動停止するガードレールを設けています。
- **負荷監視の緩和と初期化ラグ対策 (v0.10.0/v0.11.0):** 低スペック環境での安定性を向上させるため、自動停止のしきい値を緩和（200ms/20回）しました。また、Workerの初期化やJITコンパイルに伴う「起動直後の遅延」による誤作動を防ぐため、開始直後の180フレーム（約3秒）を「ウォームアップ期間」とし、この期間内はパフォーマンス監視を行わないようにしています。
- **透明なレイヤー構造による視認性確保:** 背景アニメーションを表示する `canvas` (z-index: 0) と、前面のテキストを表示する `div` (z-index: 1) が重なる構造において、前面の要素がカテゴリ色等で塗りつぶされるとアニメーションが隠れてしまいます。これを防ぐため、アニメーションがアクティブな間（`.anim-active` クラス付与時）は、前面のコンテナの `background-color` を `!important` 指定で透過（または極めて高い透明度の色に）上書きし、レイヤーの重なりに関わらずアニメーションが常に視認できるようにしています。

### 背景アニメーション：視認性向上と制作ガイド
- **理由:** 背景アニメーションと前面のテキスト（業務カテゴリ名やタイマー）が重なる際の視認性を確保するため、当初は `usePseudoSpace` と `ignoreExclusion` という 2 つの独立したプロパティを使用していました。しかし、これらは「前面 UI をどう扱うか」という一つの目的を共有しており、開発者にとって関係性が分かりにくいものでした。そこで、これらを `exclusionStrategy` という単一のプロパティに統合し、`'mask'` (デフォルト), `'jump'`, `'freedom'` の 3 つの戦略から選択する設計にリファクタリングしました。
- **FG/BG 概念と用語の採用:** 画面を「FG (Foreground: カテゴリ名やタイマーの表示帯)」と「BG (Background: アニメーションキャンバス)」という 2 つの層として定義しました。開発者向け UI では「FG回避 (FG Avoidance)」という用語を採用し、ユーザーインターフェースとしての前面要素と、表現としての背景要素の関係性を直感的に理解できるようにしています。
- **制作ガイド (Creative Guide) の役割:** 単なる技術仕様書ではなく、技術的な制約（マスキング等）を「クリエイターによる表現の選択（戦略）」としてポジティブに捉え直すためのガイドとして再編しました。これにより、制作の楽しさを高めつつ、実用的な視認性を確保することを狙っています。

### QL-Animation Studio (β版)
- **開発目的:** ユーザーコミュニティによるアニメーション提供を促進し、製品の「Visual Healing」体験を拡張するためです。
- **サンドボックスモデル:** 開発者が安心して実験できるよう、アニメーションロジックは Web Worker で実行し、メインスレッドの DOM やストレージ、およびネットワークへのアクセスを物理的に遮断しています。
- **β版の明示:** 実装が進行中であり、API の破壊的変更の可能性があることを示すため、一貫して「β版」と表記します。
- **開発・検証用モジュール (devOnly):** パッケージの肥大化を防ぎ、かつ開発時の確実な検証を両立するため、`devOnly: true` フラグを持つモジュールを導入しました。これらは一般配布用の ZIP パッケージからは物理的に除外され、製品版のアプリ設定メニューにも表示されません。
- **教育用サンプルモジュールの標準:** ジュニアエンジニアのモデルとなるサンプル（`wind_tunnel.js` 等）には、日英バイリンガルの JSDoc、マジックナンバーの排除、`devOnly: true` の設定、および全サポート言語のメタデータを含めることを標準としています。
- **Studio UI の視認性:** サンプル選択ドロップダウンにおいて、製品版モジュールには 📦、開発用には全角スペースを付加することで、垂直方向の整列と種別の識別を容易にしています。
- **開発体験の最適化 (Single Development Path):** 当初、Animation Studio にコード断片を挿入する「マジック・スニペット」機能の導入を検討しましたが、既存のアニメーションをサンプルとして読み込んで修正するパスと、スニペットから構築するパスの 2 つが存在することで、開発者の迷いを増やす懸念が生じました。そこで「サンプルをベースに開発する」という単一のパスに集中させる方針を決定しました。応用可能なテンプレートコードは、製品版には含まれない `devOnly: true` のアニメーションモジュールとして提供することで、実例を通じた学習と開発を促進します。
- **コードパーサーの最適化:** Studio でのファイル読み込み時、`metadata` や `config` をクラスボディから積極的に除去することで、エクスポート時の重複定義を防止しています。
- **モーダル UX 標準:** 背景クリックまたは「閉じる」ボタンで閉じられること、およびリンクの視認性（プライマリカラーと下線）を統一しています。

### ナビゲーションとポータビリティ
- **パス・ポータビリティ標準 (Path Portability Standard):** ローカル開発（Python server 等）、Vercel デプロイ、およびパッケージ化された Chrome 拡張機能のすべての環境でアセットが正しく読み込まれるようにするためです。HTML 内のアセット参照（CSS, JS, 画像）や内部リンクは、先頭にスラッシュを付けない「相対パス」（例: `shared/css/...` や `../app/app.html`）を徹底します。
- **言語設定の永続化標準 (Language Persistence Standard):** ランディングページとサブプロジェクト（Studio, Editor, Guide）間で表示言語を同期するため、URL パラメータ `?lang=` を活用します。初期化時はブラウザのロケールよりもこのパラメータを優先し、言語変更時は `window.history.replaceState` を用いて URL 状態を維持します。
- **Vercel デプロイ時のアセット・ルーティング:** Vercel 上でのシンボリックリンクの欠如を補完するため、`vercel.json` にて `/(.*)/shared/(.*)` を `/shared/$2` へマッピングする再帰的な書き換えルール（Rewrite）を適用しています。これにより、サブディレクトリ（`/studio/` 等）からの相対的なアセット要求をルートの `shared` ディレクトリへ透過的にリダイレクトします。

### 二系統ビルド戦略 (Dual-Build Strategy)
- **理由:** ストア配布用の安定版（Release）と、新機能の先行検証やデバッグを目的とした開発版（Dev）を明確に区別するためです。
- **Release版:** 青色アイコン。`devOnly: true` が設定されたアニメーションモジュールは物理的に除外され、パッケージサイズを最小化します。
- **Dev版:** オレンジ色アイコン。すべての開発用アニメーションを含み、マニフェスト名に `(Dev)` サフィックスが付与されます。

### 文字列の永続性とローカライズ・ポリシー
- **理由:** IndexedDB に保存されるシステム予約文字列（`SYSTEM_CATEGORY_IDLE` 等）は、将来的なデータの互換性を維持するために変更してはなりません。また、多言語対応を円滑にするため、特定の言語に依存しない値（例: `__IDLE__`）を内部的な ID として採用し、UI 表示の際のみ翻訳関数（`t()`）を通してローカライズします。

### UI レイアウトの堅牢性: 可変長テキストへの対応
- **理由:** 多言語対応に伴い、ボタンやラベルのテキスト長が大きく変動します。レイアウト崩れを防ぐため、重要な UI 要素には `min-width: 0`, `overflow: hidden`, `text-overflow: ellipsis` を適用し、どのような言語でも「壊れない」UI を維持します。

### UI インタラクション: フォールスルー・パターン
- **理由:** 背景アニメーションへの干渉（Cats への給餌クリック等）と、前面テキストの選択・操作を両立させるためです。タイマー表示コンテナ（`#current-task-display-base`）には `pointer-events: none` を設定してクリックを背後のキャンバスに透過させ、操作が必要なテキストやラベルのみ `pointer-events: auto` で個別に有効化しています。

### テスト容易性: E2E セレクターの具体性
- **理由:** モダルの閉じるボタン等が複数の機能で共通のクラス（`.close-btn`）を持つと、E2E テスト（Playwright）において誤った要素を操作するリスク（Flaky Test）が生じます。そのため、機能ごとに固有のクラス（`.report-close-btn`, `.tag-aggregation-close-btn` 等）を付与し、テストの安定性を高めています。

### セキュリティ: 言語パラメータのサニタイズ (`getSafeLang`)
- **理由:** URL パラメータや設定から読み込まれる言語コードをそのまま使用すると、不正な文字列の注入や予期せぬ動作を招く恐れがあります。ホワイトリスト形式の `getSafeLang` を通すことで、許容された言語コードのみが CSS フォントスタックやドキュメントタイトルに適用されるようにしています。

### 日報出力における経過時間フォーマット
- **理由:** スペースの有無により視覚的なリズムを生み出し、10分未満の短いタスク（`1h 5m`）を際立たせつつ、長時間のタスク（`1h15m`）をコンパクトに表示するためです。

### コピー操作の標準化と視認性の確保
- 理由: ユーザーの期待に沿った一貫したフィードバックを提供するため、日報およびタグ集計のコピー操作は「トースト通知のみ（ボタンのテキストは変えない）」に統一しています。また、ホバー時に M3 のトナル・インバージョン（Primary Container 等）を適用することで、ダークテーマを含めた全ての環境で確実な視認性と「押せる感」を両立させています。

### 設定画面のアイコン化とツールチップ (v0.31.0)
- **理由:** 設定モーダル内のタブ（一般、カテゴリ、メンテナンス等）を Material Symbols のアイコンに刷新したのは、将来的な設定項目の増加に備えて水平スペースを確保し、かつ多言語環境でのラベル長によるレイアウト崩れを防ぐためです。
- **アクセシビリティ:** アイコンのみでは機能が伝わりにくい可能性があるため、`data-i18n-title` を用いた多言語対応のツールチップを実装し、マウスオーバー時に元のラベル（General, Categories 等）を表示することで視認性と操作性を両立させています。

### 業務カテゴリ・エディタのナビゲーション最適化 (v0.32.0)
- **理由:** 業務カテゴリ・エディタをウェブベース（Vercelホスト）に移行しました。これは、拡張機能本体の更新（ストア審査）を待たずにエディタの改善を迅速に届けるためです。
- **実装:** 本体アプリの設定画面から Vercel 上の URL (`https://quick-log-solo.vercel.app/category-editor`) へ遷移させます。この際、URL パラメータ `?from=app` を付与し、エディタ側の「戻る」リンクを非表示にする制御を維持しています。これにより、拡張機能という限られたコンテキストから遷移したユーザーに対し、一貫したユーザー体験を提供します。

### Vercel デプロイ制限への配慮
- **理由:** Vercel Free プランには「24時間で100デプロイ」という制限があるためです。
- **対策:** GitHub Actions において `paths` フィルタを活用し、ドキュメントのみの更新など製品動作に影響しない変更では Vercel デプロイをスキップするように最適化しています。

### 診断用統計情報の提供 (Diagnostic Statistics UI)
- **理由:** 不具合報告時にユーザーが内部状態（ログ件数やカテゴリ数）を正確に伝えられるようにするためです。
- **実装:** `projects/app/app.html` の About タブに統計情報表示用のプレースホルダ (`#stats-log-count`, `#stats-category-count`) を追加しました。これらは `projects/app/js/app.js` の `updateAboutStats` 関数によって IndexedDB から集計・表示され、全サポート言語の `messages.js` キー (`about-stats-logs`, `about-stats-categories`) によってローカライズされます。

### GitHub Issue テンプレートの整備

- **実装:** 新規インストール時、アラーム設定の 5 番目のスロットに「23:59」に「作業終了」を行う設定がデフォルトで有効化されます。これにより、ユーザーは必要に応じて時刻を調整したり、無効化したりすることが標準的な UI を通じて可能になりました。

### Mermaid 図のシンタックス: ユニークな参加者 ID の確保
- **理由:** GitHub の Markdown 表示において、同一ファイル内に複数の Mermaid シーケンス図が存在する場合、参加者 ID（Participant ID）が重複すると正しくレンダリングされない可能性があるためです。各図で `C_Disc`, `C_Inst`, `C_Loop` のように、用途に応じたプレフィックスやエイリアス（`AS`）を使い、グローバルにユニークな ID を使用することを徹底します。

### タイポグラフィ: 言語別フォント最適化と Web フォントの徹底
- **理由:** OS 標準のフォントへのフォールバック（特に日本語環境での明朝体へのフォールバック等）を防ぎ、全てのサポート言語（日・英・独・西・仏・葡・韓・中）で一貫したモダンな外観を維持するためです。
- **グリフと絵文字:** `Noto Sans Symbols` と `Noto Color Emoji` を明示的に導入することで、特殊記号や絵文字の表示をプラットフォーム間で統一し、デザインの意図を正確に伝えます。
- **動的なフィルタリング:** 設定画面のフォント選択肢を選択中の言語に最適化されたものに限定することで、ユーザーが迷わず適切なフォントを選べるようにしています。

### パフォーマンス最適化標準
- **理由:** 計算負荷の高いシミュレーションにおいて、多次元配列を避け、1次元の `TypedArray` と行優先インデックス（`y * width + x`）を使用することで、キャッシュ効率の向上と GC 負荷の低減を図っています。

### CSS ドキュメンテーション・ポリシー
- **理由:** `!important` を使用する際は、必ずその理由（例: アニメーション透過性の維持）をコメントで併記することを義務付けています。これにより、不用意な削除によるデグレードを防止します。

### 実装の原則
- **ディレクトリ構造を維持せよ:** ソースコードは `projects/app/` 、ユーティリティは `scripts/` に配置する。
- **ルートディレクトリのクリーンネス・ポリシー:** 検証目的で作成した一時的なスクリプト（`reproduce_*.py` や `verify_*.py` 等）をルートディレクトリにコミットしてはならない。作業完了後、コミット前に必ず削除せよ。これは CI 上の `scripts/check_root_files.py` によって厳格に検査されます。
- **Vanilla JS を維持せよ:** 実行時に必要な npm パッケージの追加は一切禁止する。
- **定数化を徹底せよ:** `projects/app/js/db.js` や `projects/app/js/app.js` にあるように、マジックナンバー、DOM ID、システム文字列は必ず定数として一元管理せよ。機能に密結合した定数はそのモジュール内で、共有されるものは `utils.js` 等で定義せよ。
- **HTML エスケープを徹底せよ:** `utils.js` の `escapeHtml` を使用し、ユーザー入力（カテゴリ名等）を表示する際は XSS を確実に防止せよ。
- **データ不整合を防げ:** `initDB` 内の自動修復ロジックのように、予期せぬクラッシュや中断が発生しても、データの整合性が保たれる（二重開始を防ぐ等）ように配慮せよ。

- **ユーザーとのコミュニケーションおよび PR 連絡は日本語で行え:** ユーザーとの対話（チャット）、開発に関するコミュニケーション（PR、コミットメッセージ等）は、すべて日本語で行うこと。
- **コミットメッセージ規約:** バージョン自動採番のため、[Conventional Commits](https://www.conventionalcommits.org/) に準拠せよ（例: `feat:`, `fix:`, `feat!:`, `BREAKING CHANGE:`）。
- **Why を記録せよ:** 重要な変更を行う際は、単に「何を変えたか」だけでなく、「なぜその方法を選んだか」を AGENTS.md の「判断の背景」に追記せよ。
- **バージョン管理とリリース:**
    - `projects/app/version.json`, `package.json`, `projects/app/manifest.*.json` のバージョン番号は常に一致させよ。
    - バージョン更新は `npm run version:bump` (`scripts/bump_version.py`) を使用して自動で行え。
    - CI 上の `scripts/verify_version_impact.py` により、変更内容（コミットタイプ）に応じた適切なバージョンアップが行われているか検証される。

    - **Jules による自律リリース:** ヒューマンエラー防止のため、Jules は機能追加や修正が完了した際、自律的に `npm run version:bump` を実行し、適切なタグを作成・プッシュしてリリースをトリガーすることが推奨される。

### 深い計画プロトコル (Deep Planning Protocol)
- **原則:** 実行前に「絶対的な確信」を得ること。
- **手順:**
  1. ゴールを深く理解するための対話を行う。
  2. 疑問点や前提条件を質問で確認する。
  3. 承認は質問への回答の後に行われる。
  4. `request_user_input` を活用し、計画承認後にのみ自律的な実行へ移る。

## 4. 設計原則の適用 (Design Principles)

本プロジェクトにおける各原則の体現方法は以下の通りです。

- **KISS (Keep It Simple, Stupid):** `logic.js` は DOM に依存せず、純粋なデータ処理に特化させている。
- **DRY (Don't Repeat Yourself):** UI の更新（`updateUI`）を 1 箇所に集約し、BroadcastChannel を通じた同期も最小限のロジックで行っている。
- **SLAP (Single Level of Abstraction Principle):** `app.js` のイベントハンドラ内では高度なビジネスロジックを書かず、`logic.js` や `db.js` の関数を呼び出す形式に統一している。
- **OCP (Open-Closed Principle):** カテゴリやフォント設定をデータ（配列・ストア）駆動にすることで、コードを書き換えずに設定を追加できる構造にしている。
- **YAGNI (You Ain't Gonna Need It):** 現時点で明確な要求がない機能（例: 高度なレポート機能、他サービス連携）は実装せず、コア機能のシンプルさを維持している。
